# 前端资源整理方案

## 背景

当前前端项目是 Vite + React + TypeScript，前端根目录为 `frontend`。资源主要放在 `src/assets` 和 `public` 下，但现在存在几类混乱点：

- `src/assets/icon/Logo` 存放了 logo、favicon、banner 等品牌资源，但代码里多处写成 `src\assets\Logo\logo.png`，实际目录和引用路径不一致。
- `src/assets/img` 同时放了用户头像、按钮背景、状态开关、颜色图片、聊天角色图标等不同用途资源，分类边界不清。
- `src/assets/fonts`、`src/assets/pages`、`src/assets/pages/Chat` 目前是空目录，容易让后续资源继续被随意放置。
- `src/assets/react.svg` 是 Vite 模板资源，当前业务没有引用，可以作为待清理项。
- `public/favicon.svg` 和 `public/icons.svg` 属于不会被打包 hash 的静态资源，应与 `src/assets` 中参与模块打包的资源区分管理。

## 整理目标

1. 让资源目录按用途组织，而不是按临时来源或文件格式组织。
2. 统一 React 代码里的资源引用方式，避免写 `src\assets\...` 这类浏览器不可移植路径。
3. 区分 `public` 静态直出资源和 `src/assets` 模块内资源。
4. 删除模板残留和空目录，降低后续维护成本。
5. 为新增资源建立命名、放置和引用规则。
6. 梳理 `pages` 和 `components` 的职责边界，让页面、业务组件、通用组件各自只做一类事情。

## 页面和组件分层契约

资源整理不只包含图片、字体和静态文件，也需要整理页面和组件的归属。当前 `src/pages` 里有一些文件已经承担了过多职责，例如：

- `src/pages/subPage/RoomSelect.tsx` 同时负责拉取房间、创建房间表单、筛选栏、房间列表、房间卡片和浮动按钮。
- `src/pages/room/ChatRoom.tsx` 同时负责房间请求、Socket 订阅、消息输入、消息列表、顶部房间信息和底部说明。
- `src/pages/room/AIChat.tsx` 同时负责 AI 角色选择、聊天输入、消息列表、流式响应状态和页面布局。
- `src/pages/room/AIChatMessage.tsx` 虽然在 `pages/room` 下，但它是消息展示组件，不是路由页面，应该移到 `components` 或对应业务组件目录。

后续整理时按下面的契约拆分。

### `src/pages`

只放“路由入口页面”。

负责：

- 接收路由参数，例如 `roomId`。
- 组合页面级布局和业务区块。
- 调用页面级 hook 拿数据、提交动作和 loading/error 状态。
- 处理页面跳转、路由守卫后的兜底展示。

不负责：

- 不写大段可复用 UI，例如卡片、表单、列表项、导航栏。
- 不直接堆很多输入框和按钮细节。
- 不放纯展示组件。
- 不放只被某个区块使用的子组件实现。

判断标准：

- 如果这个文件不能被路由直接访问，就不应该放在 `pages`。
- 如果 JSX 超过一个屏幕，并且里面有明显的卡片、表单、列表、头部、底部等区块，应该拆到组件。
- 如果一个区块可以用 props 描述输入输出，就应该抽成 component。

建议页面结构：

```text
src/pages/
  auth/
    LoginPage.tsx
    RegisterPage.tsx
    GatePage.tsx

  room/
    ChatRoomPage.tsx
    AIChatPage.tsx

  home/
    MainLayoutPage.tsx
    RoomListPage.tsx

  NotFoundPage.tsx
```

### `src/components`

放可以被多个页面复用，或职责明确的 UI 组件。

负责：

- 接收 props，渲染 UI。
- 通过回调把用户操作交给上层，例如 `onSubmit`、`onSelect`、`onJoinRoom`。
- 管理组件内部的临时 UI 状态，例如输入框聚焦、展开收起、动画状态。
- 保持可复用，不绑定具体路由。

不负责：

- 不直接调用 `api`。
- 不直接读写路由参数。
- 不直接持有页面级业务流程。
- 不把多个无关业务区块塞进一个组件。

建议按领域拆分，而不是继续使用 `components/index` 这种含义不清的目录：

### `src/hooks`

放复用的状态逻辑和业务流程。

负责：

- 封装页面或组件需要复用的状态逻辑。
- 封装副作用，例如请求房间列表、订阅 Socket、滚动到底部。
- 对外暴露清晰的数据和动作。

不负责：

- 不返回 JSX。
- 不写样式。
- 不依赖某个具体 DOM 结构，除非 hook 名称明确表达它是 DOM 行为 hook。

适合从当前页面中抽出的 hook：

```text
src/hooks/
  useRooms.ts
  useRoomChat.ts
  useAIChat.ts
  useAutoScroll.ts
```

### `src/api`

只放 HTTP 请求封装和接口类型附近的请求方法。

负责：

- 调用后端接口。
- 处理请求路径、参数和响应。
- 暴露语义化方法，例如 `roomApi.getRooms()`、`roomApi.createRoom()`。

不负责：

- 不处理页面展示。
- 不处理 toast、跳转、表单状态。
- 不持有 React state。

### `src/socket`

只放 Socket 连接和事件订阅的底层封装。

负责：

- 建立连接、加入房间、离开房间。
- 注册和取消注册消息事件。
- 对外提供稳定方法，例如 `joinRoom`、`sendMessage`、`onNewMessage`。

不负责：

- 不决定消息怎么展示。
- 不直接操作 React 页面状态。
- 不知道具体页面布局。

### `src/contexts`

只放跨页面共享的全局上下文。

负责：

- 用户登录态。
- 全局配置。
- 真正跨页面共享且需要 React Context 的状态。

不负责：

- 不放某个页面独有的表单状态。
- 不放可以通过 props 或 hook 局部维护的状态。

## 文件夹职责契约

| 文件夹 | 只放什么 | 只负责什么 | 不应该做什么 |
| --- | --- | --- | --- |
| `src/pages` | 路由页面入口 | 组装布局、读取路由、连接 hook 和组件 | 不放可复用组件，不写大段 UI 细节 |
| `src/components/layout` | 页面骨架组件 | Header、Nav、Sidebar、布局容器 | 不请求业务数据 |
| `src/components/auth` | 登录注册相关 UI | 表单、品牌展示、门禁输入 | 不直接处理全局登录态存储 |
| `src/components/room` | 房间列表和房间表单 UI | 展示房间、创建房间、筛选房间 | 不直接调用 `roomApi` |
| `src/components/chat` | 聊天相关 UI | 消息列表、消息项、输入框、AI 角色切换 | 不直接管理 Socket 生命周期 |
| `src/components/common` | 通用展示组件 | Loading、Empty、Error、基础按钮 | 不包含具体业务文案和接口逻辑 |
| `src/hooks` | 可复用逻辑 | 请求编排、副作用、局部业务状态 | 不返回 JSX，不写样式 |
| `src/api` | HTTP API 封装 | 请求后端、封装参数和响应 | 不处理页面交互和 React 状态 |
| `src/socket` | Socket 底层封装 | 连接、订阅、取消订阅、发送事件 | 不决定 UI 展示 |
| `src/contexts` | 全局上下文 | 跨页面共享状态 | 不放页面局部状态 |
| `src/types` | 类型定义 | 共享数据结构和接口类型 | 不写运行时代码 |
| `src/utils` | 纯工具函数 | 无副作用的数据处理、格式化、错误转换 | 不依赖 React 组件和 DOM |
| `src/assets` | 打包资源 | 图片、字体、主题资源 | 不放页面或组件代码 |
| `public` | 静态直出资源 | favicon、固定 URL 文件 | 不放业务组件依赖的普通图片 |

## 抽组件判断标准

满足任意一条，就优先考虑从 page 抽到 component：

- JSX 中出现重复结构，例如多个页面都有 logo 区域、表单输入、提交按钮。
- 一个页面里有明显的独立区块，例如筛选栏、列表、卡片、输入框、头部。
- 某段 UI 可以通过 props 描述，例如 `room`、`message`、`loading`、`onSubmit`。
- 文件已经混合了请求、状态、表单和展示，导致改 UI 时容易碰到业务逻辑。
- 组件名称可以清楚表达单一职责，例如 `RoomCard`、`ChatMessageInput`。

不建议抽组件的情况：

- 只用一次、只有两三行 JSX，并且没有独立语义。
- 抽出来后 props 比原始代码更复杂。
- 组件必须知道太多页面内部状态，说明边界还没想清楚。

## 关注点分离原则

1. 页面负责“这个路由要显示什么”。
2. Hook 负责“数据和行为怎么来”。
3. API 负责“怎么和后端通信”。
4. Socket 负责“怎么和实时连接通信”。
5. Component 负责“给定数据后怎么展示，以及把用户操作通过回调交出去”。
6. Asset 负责“静态资源归档和被引用”。

每个文件改动时，先问一句：这个文件现在做的事情是否超出了它所在文件夹的契约。如果超出，就应该移动、拆分或下沉到更合适的目录。

## 资源放置规则

### 放在 `src/assets` 的资源

满足以下条件的资源放在 `src/assets`：

- 被 React 组件或 CSS 模块直接引用。
- 希望经过 Vite 打包、压缩、hash 和缓存管理。
- 资源属于业务界面的一部分，例如 logo、头像、按钮图、主题图。

推荐引用方式：

```tsx
import logoUrl from '@/assets/brand/logo.png'

export function HeaderLogo() {
  return <img src={logoUrl} alt="Logo" />
}
```

CSS 中推荐：

```css
.chat-button {
  background-image: url('@/assets/chat/ui/bg-button-a.gif');
}
```

### 放在 `public` 的资源

满足以下条件的资源放在 `public`：

- 需要固定 URL 访问，例如 `/favicon.svg`。
- 第三方脚本或 HTML 入口直接依赖。
- 不希望被 Vite 改名或打包处理。

`public` 里的资源引用必须使用根路径：

```html
<link rel="icon" type="image/svg+xml" href="/favicon.svg" />
```

不要在组件里把业务图片随意放到 `public`，否则会绕过构建检查和 hash 缓存。

## 图标资源契约

图标属于 UI 资源的一部分，但它和位图、头像、品牌图不同：优先使用可维护的 React 图标组件，只有业务强绑定或需要像素级还原时，才把图标作为图片资源放入 `src/assets`。

### 图标库选型

项目默认图标库使用 `lucide-react`。

原因：

- Lucide 是线性描边图标，适合当前黑白极简、匿名聊天室、日系论坛风格。
- React 组件形式可以统一尺寸、颜色、描边宽度和 hover 状态。
- 比直接复制 SVG 或切图更容易被 TypeScript、测试和重构工具检查。

补充规则：

- 主体操作图标优先从 Lucide 选择。
- 如果 Lucide 没有合适图标，状态类或动态感更强的图标可以参考 Phosphor。
- 不混用多个图标库来表达同一类操作，例如“消息”不要一处用 Lucide、一处用 Tabler。
- `public/icons.svg` 只保留给必须固定 URL 访问的静态图标集，不作为业务组件默认入口。

安装方式：

```bash
npm install lucide-react
```

推荐使用方式：

```tsx
import { MessageCircle, Users, Search, Bell, Lock } from 'lucide-react'

export function RoomActions() {
  return (
    <button type="button" aria-label="查看成员">
      <Users className="h-4 w-4" strokeWidth={1.5} />
    </button>
  )
}
```

### 图标样式契约

默认参数：

```text
size: 16px 或 20px，跟随按钮密度决定
strokeWidth: 1.5
color: currentColor
opacity: 0.9
```

Tailwind 推荐写法：

```tsx
<MessageCircle className="h-5 w-5 text-zinc-100 opacity-90" strokeWidth={1.5} />
```

交互状态：

- hover 时只调整 `opacity`、`color` 或轻微 `drop-shadow`，不要改变图标实际尺寸，避免按钮抖动。
- active 状态可以跟随按钮整体缩放，不单独缩放图标。
- disabled 状态使用父级 `text-zinc-500 opacity-50` 控制，不给单个图标写分散样式。

使用组件库图标时：

- 不放进 `src/assets`。
- 不手写 `svg` 文件。
- 在使用组件内直接从 `lucide-react` 按需导入。
- 多处复用且带统一样式时，封装到 `src/components/common`，例如 `IconButton`。

使用自定义图标图片时：

- 业务界面图标放在 `src/assets`，参与 Vite 打包。
- 品牌图标放 `src/assets/brand`。
- 聊天角色、头像、身份标识放 `src/assets/chat/avatars` 或 `src/assets/chat/badges`。
- 不再新增 `src/assets/icon` 作为泛目录。
- 文件名使用小写 kebab-case，例如 `room-lock.png`、`admin-badge.png`。

禁止事项：

- 不在 JSX 里写 `src/assets/...` 或 Windows 风格 `src\assets\...`。
- 不把 Lucide 可表达的通用图标切成 png。
- 不在不同页面为同一个功能使用不同图标名称。
- 不在图标文件名里使用中文。

### 图标搜索和参考

找图标时按这个顺序：

1. 在 Lucide 图标列表搜索功能语义，例如 `message`、`users`、`lock`、`search`。
2. 如果 Lucide 不合适，再参考 IconSearch、Unicon 或 Iconify 做跨库对比。
3. 只有确认组件库无法满足时，才新增自定义图片资源。

参考站点：

- Lucide: https://lucide.dev/
- Lucide React: https://lucide.dev/guide/packages/lucide-react
- IconSearch: https://iconsearch.info/icon-search
- Unicon: https://unicon.sh/
- Iconify: https://iconify.design/

## 命名规范

1. 文件名统一使用小写 kebab-case，例如 `banner-80x15.png`、`sound-on.png`。
2. 避免中文文件名，例如把 `总图标.png` 改成更明确的 `icon-sheet.png` 或按用途拆分。
3. 动图和静态图如果表达同一对象，保持同名前缀，例如 `kanra.png` 和 `kanra.gif`。
4. 状态类资源使用 `name-state.ext`，例如 `sound-on.png`、`sound-off.png`。
5. 不再使用 `img`、`icon` 这类泛目录作为长期归档目录。

## 当前引用需要调整的位置

以下位置当前直接写了 `src\assets\Logo\...`，整理时需要改为 import 或 `@/assets/...` 引用：

- `src/pages/auth/Register.tsx`
- `src/pages/auth/Login.tsx`
- `src/pages/auth/Ddddr.tsx`
- `src/components/index/Head.tsx`

建议统一改成：

```tsx
import logoUrl from '@/assets/brand/logo.png'
import smallBannerUrl from '@/assets/brand/banner-80x15.png'
```

然后在 JSX 中使用：

```tsx
<img src={logoUrl} alt="Logo" />
<img src={smallBannerUrl} alt="smallLogo" />
```

这样可以让 TypeScript、Vite 和构建流程一起检查资源是否存在。

## 建议补充一个资源出口文件

如果某些资源会在多个组件复用，可以增加：

```text
src/assets/index.ts
```

示例：

```ts
export { default as logoUrl } from './brand/logo.png'
export { default as smallBannerUrl } from './brand/banner-80x15.png'
```

组件中使用：

```tsx
import { logoUrl } from '@/assets'
```

注意：只导出高频复用资源，不要把所有图片都塞进一个出口文件，否则会增加维护负担。
